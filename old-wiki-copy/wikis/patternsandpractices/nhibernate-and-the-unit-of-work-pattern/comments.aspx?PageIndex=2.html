By: jwadsack Posted on 12-22-2010 21:01

The UnitOfWorkFactory that is stored in the UnitOfWork must also be stored in Local.Data, rather than static or else the session is lost during HTTP handling.

You can fix that by replacing the UnitOfWorkFactory property in UnitOfWork.cs with the following:

       public const string CurrentUnitOfWorkFactoryKey = "CurrentUnitOfWorkFactory.Key";

       private static IUnitOfWorkFactory UnitOfWorkFactory {

           get {

               var unitOfWorkFactory = Local.Data[CurrentUnitOfWorkFactoryKey] as IUnitOfWorkFactory;

               if( unitOfWorkFactory == null ) {

                   unitOfWorkFactory = new UnitOfWorkFactory(Environment);

                   Local.Data[CurrentUnitOfWorkFactoryKey] = unitOfWorkFactory;

               }

               return unitOfWorkFactory;

           }

       }

By: jwadsack Posted on 12-28-2010 1:12

Ignore my last comment. That's a Bad Idea because it stores the NHibernate session across threads, which we really don't want to do.

I've worked around the intermittent problem of losing the session by exposing a property in IUnitOfWorkFactory and UnitOfWorkFactory to check for an active session:

   public bool HasSession {

     get { return _currentSession != null; }

   }

Then in UnitOfWork, change IsStarted to also check that there's an active session:

   pubic bool IsStarted {

     get { return CurrentUnitOfWork != null && UnitOfWorkFactory.HasSession; }

   }

Finally, I altered Current and CurrentSession properties to both check IsStarted and if that fails call Start();

This means that anytime you try to use the UnitOfWork it will start itself so it's critical that there's somewhere (e.g. Application_EndRequest in Global.asax.cs) that the UnitOfWork will be disposed if it has been started.
